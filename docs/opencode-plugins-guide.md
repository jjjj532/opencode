# OpenCode插件生态完全指南：从「裸机」到「一人AI团队」的进化

## 一、为什么你需要插件生态？

许多开发者认为OpenCode本身已经足够强大，安装插件完全是多此一举。然而，使用OpenCode的默认配置充其量只能得到60分。OpenCode本身是一把好剑，但它只提供剑谱，不帮你规划任务，不帮你调度代理，不帮你管理需求，更不帮你扩展技能。

插件生态的价值在于将「裸机」升级为「完整系统」。如果把OpenCode比作操作系统，那么Oh-My-OpenCode就像给系统装上多核CPU和智能调度器，OpenSpec像给系统装上项目管理软件，Superpowers则像给系统装上各种专业软件。三者各管一摊、相互配合，共同构成一个完整的AI开发环境。

## 二、三个插件的定位与职责

### 2.1 全局视图

三个插件覆盖不同层级，各有明确的分工：

| 插件 | 解决的问题 | 核心能力 | 层级 |
|------|------------|----------|------|
| Oh-My-OpenCode | 执行效率 | 多代理编排、工具增强、工作流自动化 | 编排层 |
| OpenSpec | 需求管理 | 规范驱动开发、任务分解、变更追踪 | 规划层 |
| Superpowers | 能力扩展 | 自定义技能、嵌入式MCP、权限控制 | 能力层 |

这种分层设计的核心理念是：规划层定义「做什么」，编排层决定「怎么做」，能力层提供「用什么干」，执行层处理「具体操作」。四层协同，形成完整的开发工作流。

### 2.2 层级关系详解

在最底层是OpenCode的基础工具集，包括Read、Write、Edit、LSP、MCP等底层操作能力。在此基础上，三个插件分别强化了不同层面的能力。OpenSpec作为规划层，负责在写代码之前先和AI达成共识，确保需求清晰；Oh-My-OpenCode作为编排层，负责高效调度多个代理并行工作，确保执行效率；Superpowers作为能力层，负责按需加载专业技能，确保工具完备。

## 三、Oh-My-OpenCode：一人AI团队的构建

### 3.1 Sisyphus：永不停歇的AI包工头

Oh-My-OpenCode的核心是一个名为Sisyphus（西西弗斯）的主代理系统。这个名字非常贴切——它会一直推着任务往前走，直到干完为止。

使用OpenCode传统方式的痛点很明显：丢一个复杂需求给AI，它做一半停下来等你反馈；遇到问题不会自己解决，直接摆烂；多个任务只能串行执行，一个干完才能干下一个。这种模式导致开发效率低下，开发者需要频繁介入和等待。

Sisyphus彻底改变了这一模式。以前是「需求 → AI做 → 停住 → 我反馈 → AI继续 → 又停住 → ...」的循环，现在变成了「需求 → Sisyphus调度 → Hephaestus深度执行 → Explore查代码 → Oracle咨询架构 → 全部并行 → 干完为止」的高效流程。

### 3.2 代理天团：各有所长的专业AI

Sisyphus不是一个人在战斗，它背后有一整个代理团队：

| 代理 | 模型 | 职责 |
|------|------|------|
| Sisyphus | claude-opus-4 / kimik2.5 | 主调度，规划、分解、验证 |
| Hephaestus | gpt-5.4 | 深度工作者，自主研究+端到端执行 |
| Prometheus | claude-opus-4 | 战略规划师，访谈式需求分析 |
| Oracle | 高推理模型 | 架构咨询、复杂调试 |
| Librarian | - | 文档和外部代码搜索 |
| Explore | - | 快速代码库探索 |

这种团队协作模式意味着什么？丢一个复杂需求给Sisyphus，它会自动判断应该由谁来干这个活，然后并行调度多个代理同时开工，互不耽误。以开发一个完整的React项目Google OAuth登录功能为例，Sisyphus会分解任务：后端中间件由Hephaestus负责，前端登录组件由Visual-Engineering负责，每次提交由git-master保证原子性，底层操作由OpenCode的基础工具执行。以前需要三天的工作，现在可能三小时就能完成。

### 3.3 哈希锚定编辑：解决AI改错行的问题

哈希锚定编辑是Oh-My-OpenCode最具创新性的功能之一。开发者在使用AI编程时经常会遇到这样的情况：AI读取了代码后准备在第15行插入一行，结果你或别的工具在第15行前面插入了一行——AI看到的「第15行」已经不是原来的第15行了，导致修改位置错误。

Oh-My-OpenCode通过为每一行添加内容哈希来解决这个问题。修改前的文件每一行都带有内容哈希标识，如「11#VK| function hello {」「22#XJ| return "world";」「33#MB| }」等。当AI要修改第22行时，系统会先验证这行的哈希是否还是#XJ。如果文件被外部修改导致哈希对不上，修改直接拒绝。这个简单的改动将代码编辑成功率从6.7%提升到68.3%，大幅减少了无效修改和由此引入的bug。

### 3.4 ultrawork：懒人终极命令

如果觉得上述功能都太复杂，只需要记住两个命令：

```
ultrawork
```

或者使用简写：

```
ulw
```

输入这四个字，系统自动激活所有代理，调动一切资源把任务完成。开发者可以做自己的事情，别忘了睡前检查结果就行。这种设计让AI编程变得像使用魔法一样简单，同时保留了专业用户对细节的控制能力。

## 四、OpenSpec：规范驱动开发的实践

### 4.1 模糊需求是程序员的隐形杀手

很多程序员发现AI写代码很快，但经常写到最后一对照，发现做错了。为什么会这样？原因很简单——你给的需求本身就不清楚。「帮我做个用户登录功能」「做一个聊天界面」这种模糊的需求，AI理解的和你想的完全是两码事。这就是典型的「模糊需求」问题。

OpenSpec正是来解决这个问题的。它的核心理念是「在写代码之前，先把事情做对」。通过强制性的规范文档，让开发者在动手之前先和AI达成共识，避免后期的返工和扯皮。

### 4.2 OpenSpec的工作流

使用OpenSpec的完整流程如下：用户输入「/opsx:propose 添加 Google OAuth 登录」，OpenSpec自动生成完整的规划文档结构，包括proposal.md（为什么做）、specs/（具体规范，可验证）、design.md（技术方案）、tasks.md（任务清单）。用户确认后输入「/opsx:apply」，AI开始执行任务清单，每完成一个任务打一个勾。最后用户输入「/opsx:archive」，将完成的变更归档到archive/目录，下次想回顾随时能看。

这种工作流的优势在于：规范文档是唯一的真相来源，AI严格按照文档执行，不会自由发挥导致偏差。同时，变更历史被完整记录，便于追溯和回溯。

### 4.3 实际体验对比

以给项目添加深色模式为例。传统的做法是直接让AI修改，结果做了三套方案用户都不满意，来回改了两天。使用OpenSpec后，流程变成：输入「/opsx:propose 添加深色模式」，AI生成的specs里明确定义了「切换时动画时长300ms」「localStorage存储」「支持系统偏好自动检测」——这些细节是之前没想到但实际需要的。结果一次通过，20分钟搞定。这种差异源于规范文档在动手之前就暴露了需求的模糊点，让开发者和AI能够提前对齐。

## 五、Superpowers：十八般武艺的技能系统

### 5.1 技能系统的设计理念

Superpowers借鉴了Claude Code的Skills机制，给OpenCode增加了一个技能系统。一个技能包含三个核心要素：系统指令针对特定领域的专业提示词，嵌入式MCP技能自带的能力（如浏览器自动化）按需加载，权限范围技能只能在指定范围内操作保证安全。

这种设计的核心理念是「用什么干」与「谁来干」的分离。代理解决的是「谁来干」的问题，技能解决的是「用什么干」的问题。两者可以独立配置、灵活组合。

### 5.2 开箱即用的内置技能

Superpowers提供了几个开箱即用的技能：

playwright技能用于浏览器自动化、网页测试、数据抓取。当AI需要操作浏览器时，这个技能会自动加载相应的提示词和MCP能力，让AI知道如何操作无头浏览器、如何处理反爬措施、如何等待动态内容加载。

git-master技能用于原子提交、变基操作、历史搜索。AI在进行版本控制操作时会自动使用这个技能的专业指令，确保提交规范、操作安全。

frontend-ui-ux技能用于设计优先的UI开发，不依赖设计稿。这个技能让AI能够根据描述创建美观的用户界面，而不是机械地执行指令。

### 5.3 技能的加载与使用

技能的加载是按需进行的。当用户提出一个需求时，如果没有对应技能，AI可能直接写代码，能否成功全凭运气。如果有对应技能，AI会先加载这个技能的提示词和MCP，立刻具备专业能力，知道如何正确完成任务。

这种按需加载的机制有两个优势：一是避免加载不必要的技能导致资源浪费，二是确保每次任务都使用最合适的工具。

## 六、三个插件的深度协作

### 6.1 协作流程图

三个插件的完整协作流程如下：

```
用户需求输入
    ↓
规划层：OpenSpec
  /opsx:propose → specs → design → tasks
  核心问题：要做成什么样？达成共识再动手
    ↓
编排层：Oh-My-OpenCode (Sisyphus)
  ultrawork → 代理调度 → 任务执行 → 验证结果
  核心问题：怎么高效干完？并行+自动化
    ↓
能力层：Superpowers (Skills)
  playwright / git-master / frontend-ui-ux
  核心问题：用什么能力干？技能按需加载
    ↓
执行层：OpenCode 基础工具
  Read / Write / Edit / LSP / MCP
  核心问题：具体怎么操作？底层工具支持
```

### 6.2 真实场景演练

以给React项目添加Google OAuth登录为例，展示完整的协作流程：

在规划阶段，OpenSpec生成OAuth规范、定义登录流程、Token存储方案，确保需求清晰。

在分解阶段，Sisyphus将任务分解为：后端中间件、前端组件、状态管理三个并行任务。

在执行阶段，Hephaestus负责后端中间件开发，Visual-Engineering负责前端登录组件，git-master保证每次提交都是原子性的，底层操作由OpenCode执行。

最终结果：以前一个人干三天的活，现在AI团队三小时干完。

### 6.3 兼容性说明

三个插件覆盖不同层级，大概率不会冲突。各管一摊，不抢饭碗。唯一可能的小问题是两者都用了一些hooks（会话自动化），但Oh-My-OpenCode官方已经兼容了Superpowers，不会产生冲突。

## 七、安装与配置

### 7.1 Oh-My-OpenCode安装

```
opencode
# 然后告诉它：Fetch and follow instructions from https://raw.githubusercontent.com/code-yeongyu/oh-my-openagent/refs/heads/dev/docs/guide/installation.md
```

或者使用npm全局安装：

```
npm install -g oh-my-opencode
```

安装过程会询问几个问题：有没有Claude Pro订阅？有没有ChatGPT Plus？没有的话直接选No，会使用免费的OpenCode Zen模型。

### 7.2 OpenSpec安装

```
npm install -g @fission-ai/openspec@latest
cd your-project
openspec init
```

### 7.3 Superpowers安装

```
opencode
# 然后告诉它：Fetch and follow instructions from https://raw.githubusercontent.com/obra/superpowers/refs/heads/main/.opencode/INSTALL.md
```

### 7.4 完整安装后的能力

三个都装完，你得到的是：

- OpenCode的底层能力
- Oh-My-OpenCode的执行效率
- OpenSpec的需求管理
- Superpowers的技能扩展

这才是完整版OpenCode。

## 八、使用建议与场景选择

### 8.1 增量建议

如果只想体验提升，直接装Oh-My-OpenCode。一个ultrawork改变工作方式。

如果项目复杂、经常返工，加装OpenSpec。需求先对齐，减少返工。

如果有特殊场景（浏览器自动化、Git操作、UI开发），再加Superpowers。十八般武艺，按需调用。

### 8.2 场景对照表

| 场景 | 推荐工具 | 原因 |
|------|----------|------|
| 快速原型验证 | OMO | 追求速度，不纠结质量 |
| 核心业务代码 | Superpowers | 强制TDD，保证质量 |
| 团队协作、需要文档 | OpenSpec | 规范统一、可追溯 |
| 生产级项目 | 三者协同 | 规范+流程+效率，缺一不可 |
| Bugfix | 原生OpenCode | 杀鸡不用牛刀 |

## 九、总结

插件生态将OpenCode从一个简单的编程助手升级为完整的AI开发平台。装之前，OpenCode是「裸机AI」，只能帮你写代码；装之后，OpenCode是「一人AI团队」，Sisyphus帮你推着活干完。

OpenSpec让需求从模糊到清晰——装之前需求模糊，做一半返工；装之后先签合同再干活，规范清晰。

Superpowers让能力从匮乏到丰富——装之前默认技能有限，装之后十八般武艺，按需加载。

AI不会取代程序员，但会用插件生态的人会取代不会用的。掌握这三个插件的使用，就是掌握了AI辅助开发的核心竞争力。
